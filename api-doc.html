A word of caution: The current implementation of the API is classified « unstable ». Changes in the API may (and will most certainly) occur and will be published in this documentation.

1               Introduction

The eSHAB platform allows submitting and fetching publications per REST interface. The following formats are supported:

·         JSON is used by the Web UI and is recommend for external UIs too.

·         PDF is available per single publication and publication list.

·         XML supports some more functions than JSON and was implemented especially for external data providers and consumers. The XML interface allows submitting multiple publications per once, what is not possible with JSON. In this documentation, the main focus is on the XML interface.

2               Description of the interfaces

2.1         Common properties

2.1.1     Authentication

Dependent on the request, a prior authentication is necessary. Authorization requires a user, like in the Web UI. Thus, a new user and registration office have to be registered in order to execute the following actions:

·         Submitting publications

·         Fetching non-published publications of its registration office(s)

The authentication works as follows:

1.    A POST request is sent to /api/v1/login with username and password.

2.    The response has statuc code 200 OK and contains a X-AUTH-TOKEN header. Are username and password invalid, an error code 403 FORBIDDEN is returned.

3.    Whenever a protected resource is requested, the X-AUTH-TOKEN header has to be set with the value previously returned. The token expires after 30 minutes of inactivity. In this case, the response has the 401 Unauthorized error code.

2.2          

A concrete example of a request with curl can be found below.

2.2.1     Formatting option

Whenever a request is sent, that expects XML as response format, the following request header has to be set:

 Accept: text/xml

Dependent on the interface, this might not be necessary. But it is suggested to always set it. Otherwise the right format is not guaranteed.

2.2.2     Format of a publication

A publication consists of the following elements:

<publication>
    <meta>
        ...
    </meta>
    <content>
        ...
     </content>
</publication>

 

The publication data is mainly divided into three blocks:

·         meta contains the metadata of a publication. This is the same for all publication types.

·         privatemeta  contains the private metadata of publication. Same for all publication types. If includePrivateMetadata flag was set to true then private metadata will be fetched with the requested publication.

·         content is the main content of the publication. The data structure is different dependent on the publication type.

The format of a publication looks different dependent on the use case. A date is represented in the format yyyy-MM-dd, e.g. 2017-04-03 for the third of April 2017. The time is added as follows: yyyy-MM-ddThh:mm:ss, e.g. 2017-04-03T13:00:00. Both corresponds to the ISO standard. Accordingly the timezone can be added at the end.

Are there multiple values for an attribute, the XML element is repeated according. Example: The following publicaiton has multiple secondary tenants:

<publication>
     <meta>
            <secondaryTenants>
                  <tenantCode>kabow</tenantCode>
                  <publicationDate>2017-04-15</publicationDate>
            </secondaryTenants>
            <secondaryTenants>
                  <tenantCode>kabzh</tenantCode>
                  <publicationDate>2017-04-15</publicationDate>
            </secondaryTenants>
     ...

 

Such an attribute's name is always in plural.

2.3         Fetching published publications

Endpoint: GET /api/v1/publications?publicationStates=PUBLISHED

Response: Contains the response code 100 OK and the 100 newest publications.

Parameters: The request can be extended with any combinations of the following parameters. The request parameters are attached in the form ...?publicationStates=PUBLISHED&parameter2=value2&parameter3=value3.  A new parameter always comes after the "&" character. Parameter name and value are separated by "=". A parameter name in plural can be given multiple times. Example: publicationStates=PUBLISHED&cantons=BE&cantons=ZH searches after all published publications with either canton Zurich (ZH) OR Bern (BE).

Parameter name

Description

Type

Searches for one or multiple words within the publication.

Searches for publications with the given numbers.

Error handling: When invalid parameters are given, a 400 Invalid Request Response code is returned. If the publicationStates parameter was not set correctly, a 401 Unauthorized or 403 Permission denied response might be returned, since the user can see only published publications.

2.4         Submitting publications

Endpunkt: POST /api/v1/publications/_bulk

Body: Contains one or more publications in the format <publications><publication>...</publication>...</publications>.

Response: Is the request successful, a 200 OK and the generated publications are returned. The returned publications already contain the generated data, e.g. the publication number, if it was not set manually.

Error handling: Is one or more publications erroneous, they are returned with a status code 400 Invalid Request. The error message is in English and contains the erroneous attribute. Example:

<publications>
     <publication>
            <input>...</input>
            <errorMessage>...</errorMessage>
     </publication>
     ...

2.5         Checking submitted publications

A publication originator can check the status of his publications, before they are published. The endpoint is the same as when searching for published publications, but does not need the publication state parameter.

Endpoint: GET /api/v1/publications?registrationOfficeIds=[your-id]

Parameters: Additionally to the parameters described above, publicationStates can be used with any state. The value of the state is given in the metadata table above. The response and error handling is the same as for the publication search endpoint.

3               Examples

The following examples use the command line tool curl. They also use the test environment's domain int.eshab.shab.ch. It can be replaced by the productive domain amtsblattportal.ch.

3.1         Login

curl https://int.eshab.shab.ch/api/v1/login --data "username=[your-mail-address]&password=[your-password]" -i

 

This returns the response header X-AUTH-TOKEN, which is the authentication token that expires after 30 minutes of inactivity. As long as it is active, it must be set as X-AUTH-TOKEN header with any subsequent request, or at least with those that require authentication. For instance published publicaitons do not need authentication.

In the request above, [your-mail-address] and [your-password] are replaced through the e-mail address and password of the user.

3.2         Submitting publications

curl https://int.eshab.shab.ch/api/v1/publications/_bulk -H "X-AUTH-TOKEN: [your-token]" -H"CONTENT-TYPE: text/xml" --data-binary "@publications.xml"

 

This request submits publications, that are defined in publications.xml. The filename can be different, curl just transforms the file content to the request body. Important are the request headers:

·         X-AUTH-TOKEN: Contains the token that was retrieved with the "Login" example. It grants permissions for submitting publications in the name of the logged in user.

·         CONTENT-TYPE: Must be set, so the request is interpreted as XML.

3.3         Checking submitted publications

Let us assume that we have submitted publications with publication number A and B for the registration office myOffice. These values are either sent within the XML structure or - if generated - can be read from the response. In this case, the state of the publications can be requested as follows:

curl https://int.eshab.shab.ch/api/v1/publications?registrationOfficeIds=myOffice&publicationNumbers=AW01-308814108&publicationNumbers=AW01-948856028 -H "X-AUTH-TOKEN: [your-token]" -H "CONTENT-TYPE: text/xml"
